## 0. 3대 운영 규칙

| # | 규칙 | 요약 |
|---|------|------|
| **R1** | Git 플로우 | `pull → branch → 작업 → commit → PR → auto-merge` 순서 강제 |
| **R2** | 작업 대시보드 | 작업 블록 종료마다 진척·다음할일 기록 |
| **R3** | 1작업–1정지 | 자동/권한건너뛰기 모드여도 **한 작업만 끝내고 멈춰** 보고 |

자동 모드(`--dangerously-skip-permissions`, auto-accept)에서도 R3는 무효화되지 않습니다.
"권한을 묻지 않는 것"과 "여러 작업을 연달아 하는 것"은 별개입니다.

### 표준 Git 플로우

```bash
git checkout main && git pull origin main
git checkout -b feature/worldcup/<description>
# ... 작업 ...
git add -A && git commit -m "<type>(<scope>): <subject>"
git push -u origin feature/worldcup/<description>
gh pr create --fill
gh pr merge --auto --squash --delete-branch
```

> ⚠️ Dacon Organization 룰셋: PR 제목과 브랜치의 타입 표기가 다릅니다
> (제목은 `feat:`, 브랜치는 `feature/`). 제목 수정만으로는 체크가 재실행되지 않으며
> `--admin` 우회가 불가능합니다.

## 1. 리서치 운영 프로토콜 (이 레포 고유)

기획·설계 과정에서 발생하는 불확실성은 **사용자가 아니라 에이전트가 감지하고 발주**합니다.

### 발주 트리거 — 하나라도 걸리면 원장에 적재

| ID | 트리거 | 설명 |
|---|---|---|
| **T1** | 근거 공백 | 기획서에 들어갈 수치·주장에 출처가 없다 |
| **T2** | 대안 교착 | 두 설계안이 사실관계로만 갈리는데 그 사실을 모른다 |
| **T3** | 규정 리스크 | 라이선스·저작권·심사 규칙 저촉 가능성이 감지된다 |
| **T4** | 기술 미검증 | 성능·번들 크기·브라우저 호환처럼 실측 근거가 필요하다 |
| **T5** | 반박 필요 | 심사자 예상 질문에 답할 근거가 없다 |

### 작업 블록 종료 규약

**모든 작업 블록은 끝날 때 반드시 아래 둘 중 하나를 출력합니다.**

- `📮 발주: Pn, Pn+1` — 복사해 붙일 수 있는 완성형 프롬프트를 그 자리에 제시
- `📭 발주 없음` — 원장 대기 항목이 없거나 아직 웨이브가 차지 않음

사용자가 "지금 조사할 게 있나?"를 스스로 판단할 필요가 없어야 합니다.

## 2. 절대 위반하면 안 되는 대회 규칙

| 규칙 | 결과 |
|---|---|
| 특정 선수·팀·국가 **비하·명예훼손 콘텐츠** | **심사 제외** |
| 저작권·라이선스 위반 자산 사용 | **실격 또는 감점** |
| 2026-08-03 10:00 이후 커밋 | **실격** |
| 심사자가 별도 키 없이 확인 불가한 기능 | 평가에서 제외 |
| 동적 인터랙션이 실제로 작동하지 않음 | 해당 기능 평가 제외 |

**비하 금지는 카피 한 줄 단위로 관리합니다.** 특정 인물의 판단을 "잘못"으로 규정하는 표현을
쓰지 않고, "다른 선택지도 있었다"는 관점으로 서술합니다.

## 3. 코딩 규칙

- **언어**: 대화·주석·커밋 메시지·문서는 **한국어**. 변수/함수명만 영어 CamelCase
- **들여쓰기**: 2 spaces
- **TypeScript**: strict mode, `any` 금지
- **Next.js**: App Router, Server Component 우선
- **반응형**: Mobile First 필수 — 심사자가 모바일로 열 가능성을 전제
- **에러 처리**: try-catch는 단순 로그가 아닌 적절한 에러 응답 반환
- **날짜/시간**: KST 기준
