# 분석 노트북 서사 v1.0 — 9장 구조와 기획서 절 매핑

> [ML_설계_v1_0.md](ML_설계_v1_0.md)의 학습 파이프라인을 **Jupyter 노트북 서사**로 구현하는
> 규약입니다. 사용자 확정 선호: 분석은 스토리가 있는 노트북으로 — **py 스크립트만 뽑는 방식
> 금지**. 기획서 10절 "분석 노트북 서사 연결" 확정 사항의 설계 정본.

## 1. 왜 노트북인가 — 서사 원칙

파이프라인을 py 스크립트 묶음으로 만들면 실행은 되지만 **읽히지 않습니다**. 이 프로젝트에서
노트북은 세 역할을 동시에 합니다:

1. **학습 파이프라인의 실행체** — 노트북 순차 실행이 곧 파이프라인 실행 (별도 스크립트와의
   이중 관리 금지. 유틸 함수의 `src/analysis/` 모듈 분리는 허용하되, 실행 진입점은 노트북)
2. **기획서 10절의 증거 공장** — 각 장의 그림·표가 원고에 그대로 인용됨 (§2 매핑)
3. **심사자·독자가 읽는 문서** — 각 장 서두에 "이 장의 질문", 말미에 "이 장의 답"을 마크다운
   셀로 서술. 코드보다 서사가 먼저 (분량이 아닌 커버리지 원칙, ADR-000과 동일 철학)

## 2. 9장 구조 총괄표

파일 위치·명명: `notebooks/01_eda.ipynb` ~ `09_report.ipynb`, 그림은 `notebooks/figures/`
`[설계 결정]`

| 01 | EDA | 이 데이터는 무엇을 담고 있고 무엇이 없는가 | 대회별 득점 분포·승무패 비율·**전술 컬럼 부재 확인** | 9절(데이터)·10절 |
| 02 | FE | 킥오프 이전 정보만으로 무엇을 만들 수 있는가 | 피처 테이블·Elo 궤적 그림·**리키지 기계 검증 통과 로그** | 10절 |
| 03 | LightGBM | 기준선(로지스틱)을 GBDT가 이기는가 | 기준선 대비 valid RPS 표 | 10절(모델 비교) |
| 04 | XGBoost | 같은 피처에서 구현체 차이는 얼마인가 | 03과 동일 형식 비교 행 추가 | 10절(모델 비교) |
| 05 | CatBoost | 3종 중 무엇이 이 데이터에 맞는가 | 3종+기준선 종합 비교표 | 10절(모델 비교) |
| 06 | Optuna | 튜닝은 얼마나 이득인가 | 최적화 이력 곡선·최적 파라미터 표 | 10절(**검증 곡선**) |
| 07 | 앙상블 | 결합이 단일 최량을 이기는가 + 스코어 모델과 정합하는가 | 가중·성능 표 + **bivpois λ·DC τ 학습 + TV 정합 지표** | 10절 |
| 08 | SHAP | 모델은 무엇을 보고 판단하는가 | **피처 중요도 그림**·SHAP summary | 10절(**피처 중요도**) |
| 09 | 보고서 | 방어선을 지켰는가, 브라우저까지 안전한가 | test 1회 평가·방어선 대조표·ONNX 정합 검증·**민감도 캘리브레이션**(피처_정의서 §4.5) | 10절(방어선 표)·15절(리스크) |

**해설 — 서사의 뼈대**: 01~02가 "재료의 정직한 검토"(전술 컬럼 부재를 여기서 스스로
보여줌 — ADR-008의 근거가 노트북 안에서 재현됨), 03~06이 "기준선을 이기려는 시도의 기록",
07~08이 "결합과 해석", 09가 "약속(방어선)과의 대면"입니다. 기획서 10절이 예고하는
"피처 중요도·검증 곡선"은 08·06장의 산출물이 1:1로 공급합니다.

## 3. 장별 카드

각 카드의 **수용 기준을 통과하지 못하면 다음 장으로 넘어가지 않습니다** (게이트).

### 01 EDA
- **목적**: 데이터 실체 파악과 한계의 선제 공개
- **입력**: `jfjelstul/worldcup` CSV (matches·team_appearances·tournaments·goals)
- **산출물**: 대회별 경기 수·득점 분포(포아송 적합 시각 대조 — 07장 λ 모델의 복선), 승/무/패
  기저율, 결측·이상치 목록, **"전술 성향 컬럼 부재" 명시 셀**
- **수용 기준**: **Given** 로드 완료 **When** 마스킹 셀 실행(§4) **Then** 이후 셀에서 실명
  컬럼 접근이 불가능하다 · 기저율 표가 산출된다

### 02 FE
- **목적**: [피처_정의서 §2~§3](피처_정의서_v1_0.md) 구현 + 리키지 기계 검증
- **산출물**: 피처 테이블(parquet), 주요국 Elo 궤적 그림, 공백 감쇠 적용 팀 목록(FT-R7),
  [평가_설계 §3](평가_설계_v1_0.md) 리키지 체크리스트 4항의 assert 통과 로그
- **수용 기준**: **Given** 피처 테이블 **When** 리키지 assert 실행 **Then** 4항 전부 통과

### 03~05 LightGBM / XGBoost / CatBoost
- **목적**: 동일 피처·동일 분할에서 3종 GBDT를 로지스틱 기준선과 대조
- **산출물**: 장마다 동일 형식의 valid RPS·정확도 행 — 05장 말미에 종합 비교표
- **수용 기준**: **Given** 각 모델 학습 **When** valid 평가 **Then** 기준선과의 차이가
  표로 기록된다 (이기든 지든 — 지면 §7.1 폴백 서사의 근거가 됨)

### 06 Optuna
- **목적**: RPS 목적함수 튜닝 (모델당 100 trial, ML_설계 §4.1)
- **산출물**: 최적화 이력 곡선(기획서 "검증 곡선"), 최적 파라미터 표, 튜닝 전후 대비
- **수용 기준**: **Given** 탐색 완료 **Then** best trial의 valid RPS가 튜닝 전 이하다
  (개선 없으면 튜닝 전 파라미터 유지를 명시 기록)

### 07 앙상블 + 스코어 모델
- **목적**: 검증 RPS 가중 소프트 보팅 확정 + bivpois λ·DC τ 학습 + 두 모델 정합
- **산출물**: 앙상블 가중·성능 표, λ 회귀 계수·ρ 추정치, **TV 정합 지표** (평가_설계 §2.4)
- **수용 기준**: **Given** 두 모델 **When** valid 전 경기 TV 계산 **Then** 평균 ≤ 0.05
  (초과 시 EV-R6 경로 기록)

### 08 SHAP
- **목적**: 모델 판단 근거의 해석 — "Elo 차이가 지배하는가, 폼이 기여하는가"
- **산출물**: SHAP summary(기획서 "피처 중요도"), 주요 피처 의존도 그림
- **수용 기준**: **Given** 앙상블 멤버 **Then** 전 멤버의 중요도 순위가 산출되고 상위
  피처가 도메인 상식과 모순되면 그 사실을 서술로 기록한다 (숨기지 않음)

### 09 보고서
- **목적**: 약속과의 대면 — test 1회 평가·방어선 대조·배포 안전성·민감도
- **산출물**: 방어선 대조표(측정 조건 병기, EV-R1), ONNX 정합 검증 결과(ML-R5), 슬라이더
  민감도 곡선과 δ·κ 캘리브레이션 결과(FT-R5), 폴백 발동 시 그 이력(ML-R4·EV-R3)
- **수용 기준**: **Given** 09장 완료 **Then** 기획서 10절이 인용할 모든 수치·그림이 이 장의
  산출물 경로에 실존한다

## 4. 실명 마스킹 경계 — 커밋되는 노트북은 공개 문서다

노트북은 저장소에 커밋되므로 **공개 문서**이며, "실명은 파이프라인 내부까지"(P7·P10)의
'내부'가 아닙니다. 구조적 차단 장치 ([ML_설계 §3.3](ML_설계_v1_0.md) ③):

1. **마스킹 셀**: 01장 서두, 데이터 로드 직후 — 선수명 계열 컬럼(`player_name`,
   `given_name`, `family_name` 등)을 즉시 drop하거나 해시 ID로 치환. 이후 전 장은 마스킹된
   데이터프레임만 참조 (원본 변수는 `del`)
2. **허용 표기**: 국가명·팀 코드는 허용 (P7 — 국가명 표기 허용)
3. **실명 취급 작업의 격리**: 표시층 프로파일 산정·가공명 배정(실명 필요)은 노트북이 아니라
   **비커밋 로컬 스크립트** (ADR-005 배정표 미커밋과 동일 구획)
4. **기계 검증**: 커밋 전 실명 검사 스크립트(가공명_체계 §3 검증 계열)가 노트북 출력 셀
   텍스트를 검사 — 검출 시 커밋 차단 (NB-R4)

**설계 근거** — "조심한다"는 사람의 주의력에 의존하는 방어이고, "마스킹 셀 이후 실명 변수가
존재하지 않는다"는 구조의 방어입니다. 스키마에서 실명 필드를 없앤 데이터_설계 §2.1과 같은
철학을 노트북에 적용한 것입니다.

## 6. 요구사항 (EARS)

| ID | 패턴 | 요구사항 |
|---|---|---|
| NB-R1 | Ubiquitous | 01장 마스킹 셀 이후 어떤 셀도 실명 컬럼에 접근할 수 없다 (변수 자체가 부재) |
| NB-R2 | WHEN | **WHEN** 각 장이 완료되면, 산출물(그림·표)은 `notebooks/figures/` 고정 경로에 저장된다 (기획서 인용 경로 안정성) |
| NB-R3 | Ubiquitous | 전 장은 고정 시드로 실행되며 재실행 시 09장 보고 수치가 재현된다 |
| NB-R4 | IF-THEN | **IF** 커밋 전 실명 검사가 노트북 출력에서 실명을 검출하면, **THEN** 커밋을 차단하고 해당 셀 출력을 마스킹 후 재실행한다 |
| NB-R5 | IF-THEN | **IF** 장을 건너뛰어 중간 산출물이 없으면, **THEN** 명시적 에러로 중단한다 (조용한 재계산·암묵적 의존 금지) |
| NB-R6 | IF-THEN | **IF** 재실행 수치가 커밋된 보고서와 불일치하면, **THEN** 원인(환경·시드·데이터 변경)을 규명해 09장에 기록하기 전까지 보고서를 갱신하지 않는다 |

## 7. 수용 기준 (Given-When-Then)

- **Given** 클린 클론 + 고정 환경 **When** 01~09 순차 실행 **Then** 오류 없이 완료되고 09장 수치가 커밋본과 일치한다
- **Given** 커밋된 노트북 전체 **When** 출력 셀 실명 전수 검색 **Then** 0건이다
- **Given** 기획서 10절 원고 **When** 그림·수치 인용 검증 **Then** 인용 대상 전부가 §2 매핑의 장 산출물에 실존한다
- **Given** 각 장 말미 **Then** "이 장의 답" 마크다운 셀이 존재한다 (서사 원칙 검수)
