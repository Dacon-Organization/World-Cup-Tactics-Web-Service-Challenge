우선순위는 2절의 약속 범위 그대로다: **1순위(보드+예측)의 수용 기준은 타협 불가**, 일정
압박 시 3순위 → 2순위 순으로 축소한다. 축소가 발생하면 해당 기능은 기획서가 아니라
저장소 README에 상태를 명시한다 — 기획서에 쓴 것과 구현이 어긋나는 상황을 만들지 않는다.

## 리스크 → 강등 계획

| 리스크 | 미리 정한 대응 | 근거 |
|---|---|---|
| 선례 없는 결합 — 벤치마크 전무 (P1의 이중 함의) | 시뮬레이션을 빠른 모드(5,000회) 기본으로, 정밀(25,000회)은 옵션 — 저성능 기기에서 반복 수 자동 강등 + 정직 표기 | P1·P12 |
| 모델 성능이 방어선(RPS 0.19~0.22) 미달 | 3단 대응: 피처 축소 → 모델 단순화(앙상블→단일→기준선) → 그래도 미달이면 **측정값을 그대로 공개**하고 한계 서술 강화 | P12 |
| 브라우저 추론 실패 (wasm 미지원 등) | 순수 JS 통계 폴백 + "간이 추정" 배지 — 예측 패널은 어떤 환경에서도 비지 않음 | P4 |
| 공유 API 미지원 환경 | 3계층 폴백(공유 시트 → 클립보드) — 12절 | P8 |
| OG 카드 생성 실패 | 기본 브랜드 카드 폴백 — 본 기능 무영향 | P8 |
| 시뮬레이션 장시간 소요 | 10초 임계에서 진행률+취소 상시 제공 | P5·P12 |

리스크 표의 원칙은 하나다: **실패가 기능의 소멸이 아니라 품질의 강등이 되게 한다.**
어떤 실패 경로에서도 심사자 화면이 비거나 멈추지 않는다.
