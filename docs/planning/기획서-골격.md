## 설계 원칙

1. **심사 기준 역매핑** — 모든 절은 [대회요강](../대회요강.md)의 심사 항목(참신성 30 · 감동 경험 25 · 완성도 25 · 기획/구현 일관성 20) 중 무엇을 방어하는지 명시한다. 방어 대상이 없는 절은 넣지 않는다.
2. **수치 역추적** — 기획서의 모든 수치는 원장 `회수됨` 항목(Pn)으로 역추적 가능해야 한다. 역추적이 안 되는 수치는 뺀다.
3. **약속 범위 고정** — 기획/구현 일관성 20점은 "기획서에 쓴 것과 구현이 어긋나면" 감점이다. 과한 약속을 쓰지 않고, 2절에서 약속 범위를 명시적으로 고정한다.
4. **실격 방어는 배점 이전의 전제** — 비하 금지(심사 제외)·라이선스(실격/감점)는 점수를 다투기 전에 통과해야 하는 관문. 9·13절과 전 절 카피 검수로 관리한다.
5. **원장 상태 전이는 원고 작성 시** — `회수됨`→`반영됨` 전이는 각 절 원고가 실제 작성될 때 수행한다. 골격 단계에서는 하지 않는다.

## 절별 카드

각 카드는 원고 작성 시 그대로 절의 요구 명세가 됩니다.
**"확정 사항"의 모든 수치·주장에는 원장 ID(Pn)가 붙어 있습니다** — 원고에서 이 역추적을 유지합니다.

### 1절 — 표지 (0.5p)

- **핵심 메시지**: 서비스명 + 한 줄 카피로 컨셉 즉시 전달
- **확정 사항**:
  - FIFA 공식 IP(엠블럼·워드마크·트로피) 일절 금지. "World Cup™" 단독 표기도 등록 워드마크 (P7)
  - "2026 월드컵" 서술적 언급은 조건부: 1회성 + 비공식 고지 + 반복 배치 금지 (P7)
  - 표지 그래픽은 generic football 이미지·자체 SVG만 — FIFA §5가 스스로 권장하는 방식 (P7)
- **시각자료**: 표지 키 비주얼(자체 제작 전술보드 모티프)
- **금지·주의**: 서비스명에 FIFA·월드컵 워드마크 요소 포함 금지

### 2절 — 한 장 요약 (1p)

- **핵심 메시지**: 문제 → 해법 → 차별점 → 검증 계획을 한 페이지에 압축. **이 페이지에 쓴 것만 약속한다**
- **확정 사항**:
  - 심사 기준 역매핑 미니 표(위 총괄표 축약본) 포함 — 심사자가 "이 팀은 심사 기준을 알고 썼다"를 즉시 인지
  - 구현 약속 범위 명시: 핵심 3기능(전술보드 D&D · 온디바이스 예측 · 리플레이)과 부가 기능(공유)을 구분 표기
- **시각자료**: 4블록 요약 다이어그램
- **금지·주의**: 여기 없는 기능을 뒤 절에서 약속하지 않는다 (기획일관성 20 방어)

### 3절 — 문제의식: "10위, 그리고 8위의 컷" (0.5p)

- **핵심 메시지**: 한국은 조 3위 12팀 중 **최종 10위**로 8위 진출선에서 탈락했다. "다른 선택지가 있었다면?" — 이 질문이 서비스의 존재 이유
- **확정 사항**:
  - 1승 2패 승점 3 · 골득실 −1 · 득점 2 → 조 3위 12팀 중 10위, 8위 진출선 탈락 (P6·P11)
  - 한국보다 낮은 팀은 스코틀랜드(골득실 −3)·우루과이(승점 2)뿐 (P6)
  - 전 경기 멕시코 개최 (P11)
- **시각자료**: 조 3위 12팀 순위표(8위 컷 라인 강조)
- **금지·주의**: **비하 금지 최중요 절.** 특정 인물(감독·선수)의 판단을 "잘못"으로 규정하는 표현 금지. "결과는 아쉬웠지만 다른 선택지도 있었다" 관점만

### 4절 — 서비스 개요 [필수①] (1p)

- **핵심 메시지**: 전술을 만지면(D&D) 승부가 바뀐다(온디바이스 ML) — 그리고 실제 3경기로 되돌아가 본다(리플레이)
- **확정 사항**:
  - 3대 기능 축: ① 드래그앤드롭 전술보드 ② 온디바이스 ML 승부 예측 ③ 한국 3경기 리플레이 모드
  - 대회요강 예시("드래그앤드롭 전술 보드" + "내가 감독이었다면 시나리오 시뮬레이션")와의 정합 명기
  - 시장 공백 포지션 한 줄 예고 (P1 — 상세는 5절)
- **시각자료**: 3기능 축 개념도
- **금지·주의**: 기능 나열이 아니라 "감독 경험"이라는 단일 경험으로 묶어 서술

### 5절 — 경쟁 분석과 차별화 (1p)

- **핵심 메시지**: 조사된 36+개 서비스 어디에도 "개별 선수 배치 변경 → 즉시 승부 예측"이 없다
- **확정 사항**:
  - 지정 6개(buildlineup.com, chosen11.com, lineup-builder.co.uk, footballuser.com, tactical-board.com, sharemytactics.com) 모두 예측 기능 없음. 단 lineup-builder.co.uk "38-0-0" 모드는 능력치 단순합산 추산 존재 — 과장 없이 각주 처리 (P1)
  - 인접 30여 개(FPL 도구·베팅 시뮬레이터·HF Space 데모)의 예측은 조작 단위가 "팀 선택/팀 레이팅 슬라이더"에 그침 → 개별 선수 배치·포메이션과 결합된 예측은 부재 (P1)
  - 반박 방어: 라인업 빌더류가 예측을 안 넣은 이유는 기술 제약이 아니라 "이미지 소셜 공유" 사업 모델 때문으로 추정 → 기술적으로 불가능한 영역이 아님 (P1)
- **시각자료**: 경쟁 포지셔닝 맵 (X: 조작 밀도, Y: 예측 지능)
- **금지·주의**: 경쟁 서비스 폄하 표현 금지 — "공백"으로만 서술. [추정] 표시된 결론은 기획서에서도 추정임을 유지

### 6절 — "감독 경험" 설계 의도 [필수②] (1p)

- **핵심 메시지**: 감독 경험 = 조작감(100ms) + 감독의 언어(슬라이더) + 빈 화면 없는 첫 만남
- **확정 사항**:
  - NN/g D&D 패턴: grab-handle + hover 커서, 고스트/드롭섀도 피드백, ~100ms 재배치 애니메이션, 스냅 마그네티즘, mouseup+Esc 취소 (P5)
  - 이분법 토글 대신 **연속 슬라이더**가 감독 행위감을 살림 — 라벨은 "라인 높이/압박 강도" 감독 용어 + "낮음/보통/높음" 질적 라벨 병기 (P5)
  - Nielsen 지연 임계값: 100ms(직접 조작감) / 1초(사고 흐름) / 10초(주의 유지) (P5)
  - 온보딩: 점진적 공개(슬라이더 4개+예측 패널만 노출) + 첫 진입 시 기본값으로 이미 예측치 표시 — 심사자가 빈 화면이 아닌 완성 상태에서 1~2초 내 "조작→반응" 발견 (P5)
- **시각자료**: 경험 설계 원리 다이어그램(조작→피드백 루프)
- **금지·주의**: 기능 명세가 아니라 **경험 설계 원리**로 서술 (심사 요소가 "몰입감·조작의 직관성·UX 흐름"으로 명시됨)

### 7절 — 페이지 구성과 와이어프레임 [필수③] (2p)

- **핵심 메시지**: 3페이지(전술보드 / 리플레이 / 공유 랜딩) 구조 — 각 페이지의 역할과 화면을 함께 제시
- **확정 사항**:
  - IA(정보 구조) 다이어그램 + 페이지별 모바일 우선 와이어프레임 (P5)
  - Mobile First 필수 — 심사자 모바일 열람 전제
  - 공유 수신자 랜딩: URL의 전술 상태를 읽어 즉시 렌더 (P8 — 상세는 12절)
- **시각자료**: IA 다이어그램 1 + 페이지별 와이어프레임(모바일/데스크톱)
- **금지·주의**: 와이어프레임 내 선수명 표기도 가공명 사용 (P7 — 기획서 자체도 사용자 노출 표면)

### 8절 — 핵심 인터랙션 명세 [필수④] (1.5p)

- **핵심 메시지**: 인터랙션마다 상태 전이·접근성 대안·지연 예산을 명세 — "확실히 작동"을 설계로 보증
- **확정 사항**:
  - D&D 상태 전이도: idle → grab → drag → (drop | cancel) 각 상태의 시각 피드백 명세 (P5)
  - **WCAG 2.5.7 Dragging Movements(AA)**: 모든 드래그에 클릭-클릭 대안 필수. 키보드 경로(Tab→Space→방향키→Enter→Esc)는 독립 요건 (P5)
  - 모바일: 터치 타겟 1cm×1cm, 피드백이 손가락에 가리지 않게 오프셋 (P5)
  - 지연 예산표: 슬라이더 시각 프리뷰 100ms 내 / 몬테카를로 10초 초과 시 진행률+취소 제공 (P5)
  - 확률 표기: 단일 수치 단독 금지 — 10×10 아이콘 배열 병행, HOPs 400ms/프레임 옵션, Wilson/SE는 순수 JS 즉시 계산 (P12)
- **시각자료**: 인터랙션 상태 전이도 + 지연 예산표
- **금지·주의**: 여기 명세한 인터랙션은 전부 구현 대상 — 구현 불확실한 것은 명세에서 뺀다 (동적 인터랙션 미작동 = 해당 기능 평가 제외)

### 9절 — 데이터 활용 방식 [필수⑤] (1p)

- **핵심 메시지**: 2계층 데이터 전략 — 학습은 실데이터로 정확하게, 노출은 가공명으로 안전하게
- **확정 사항**:
  - **학습층**: `joshfjelstul/world-cup-database` (Kaggle, CC BY-SA 4.0, 1차 출처). 실선수명은 모델 학습·집계 파이프라인 내부까지만 (P2·P10)
  - StatsBomb 재업로드 2건 채택 금지 — 카드 표기와 원 계약(비상업·재배포 금지)의 문구 단위 불일치 확정 (P10)
  - **표시층**: 완전 가공명 더미 데이터를 다출처 대조로 직접 구성(단일 소스 체계적 추출 금지 — DB권 리스크), 처음부터 JSON 정적 파일(`src/data/players.json` + TS 타입) (P10 + 대회요강 171~172행 권장 정합)

### 10절 — ML/DL 아키텍처 (1.5p)

- **핵심 메시지**: 직접 학습한 경량 모델을 심사자의 브라우저에서 돌린다 — 서버도, API 키도 없이
- **확정 사항**:
  - 노선: 직접 학습 경량 GBDT/로지스틱 → ONNX 변환 → 브라우저 ORT-Web(wasm) 온디바이스 추론 (P3·P4)
  - 파인튜닝 노선 기각 사유: "라이선스는 파인튜닝으로 세탁되지 않는다" (P9)
  - 성능 방어선: **RPS 0.19~0.22 / 정확도 0.50~0.55** — 공개 사례(538·Opta·학술) 정합 범위. arXiv RPS 0.127(소표본 재적합)은 벤치마크 인용 금지 (P12)
  - 몬테카를로 시뮬레이션: 정밀 25,000회 / 빠른 5,000회 (P12)
  - **분석 노트북 서사 연결**: EDA → FE → LightGBM → XGBoost → CatBoost → Optuna → 앙상블 → SHAP → 보고서. 기획서에는 이 서사의 산출물(피처 중요도·검증 곡선)이 어느 절에 인용될지 예고
  - 모델 한계 정직 고지: 확률은 단정적 예측이 아님 — 한계를 먼저 밝히는 것이 신뢰도를 높인다 (README 원칙)
- **시각자료**: ML 파이프라인 다이어그램(학습→ONNX→브라우저) + 성능 방어선 표
- **금지·주의**: 역추적 불가한 성능 수치 인용 금지. "예측한다"가 아니라 "확률을 제시한다"로 서술
