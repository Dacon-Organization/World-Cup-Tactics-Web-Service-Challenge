## 문서 운영 규칙

1. **버전 체계** (전역 §7.1 정합): 도메인별 `version{X.Y}` 폴더가 정본. **작업 중 버전업 금지** — 변경은 [_changes/vNEXT-변경노트.md](_changes/vNEXT-변경노트.md)에 누적, 마일스톤 완료 시 일괄 버전업(전체+요약본+변경사항 3종 세트, 구버전은 `_archive/`로)
2. **기능 문서 템플릿** ([ADR-000](decisions/ADR-000-문서체계.md)): gc-dating-app 7-Step × spec-kit × EARS × GWT 융합 — 8섹션 + 말미 체크리스트
3. **근거 태그**: `(Pn)` 원장 역추적 / `ADR-n` 설계 재량 / `[설계 결정]` 국소 재량 / `[추정 설계목표]` QA 실측 / `[NEEDS CLARIFICATION]` 미해소(구현 전 게이트)
4. **research 폴더는 이 체계 밖** — 원장([RESEARCH-LOG.md](../research/RESEARCH-LOG.md))이 자체 인덱스이며 상대 링크가 역추적 증거망이므로 재편하지 않음
5. **폴더 README 갱신 정책**: 매 작업마다 갱신하지 않는다. **구조 변경·대규모 리팩토링이 있는 블록에서만** 해당 폴더 README를 함께 갱신 (토큰 효율과 문서 신선도의 균형)
6. **작성 원칙 — 쉽고 상세하게**: 다이어그램에는 풀어쓴 해설("무엇을/왜/어떻게")을 병기하고, 설계 결정마다 근거 문단을 둔다. 문서가 그 자체로 학습·회고 자료가 되게 한다
