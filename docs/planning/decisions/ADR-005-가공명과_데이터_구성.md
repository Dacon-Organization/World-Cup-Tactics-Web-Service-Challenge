# ADR-005. 가공명 배정 방식과 데이터 구성 — 무작위 배정 + 프로파일 직접 산정

- **Status**: Accepted (2026-07-22)
- **관련**: [가공명_체계_v1_0.md](../data/version1.0/가공명_체계_v1_0.md) · [데이터_설계_v1_0.md](../data/version1.0/데이터_설계_v1_0.md) · P7·P10

## Context

리플레이 모드(F10)는 실제 경기 이벤트(득점자·교체)를 표시해야 하므로 실제 선수와 화면 표기의
연결이 필요하다. 그러나 P7 결론(2010카합245: 변형해도 연상되면 침해)에 따라 화면에는 완전
가공명만 허용된다. "어떻게 연결하되 연상 경로를 만들지 않는가"가 결정 대상이다.
또한 선수 능력 프로파일을 어디서 가져올지도 DB권 리스크(P10)와 얽혀 있다.

## Decision

1. **실명↔가공명 1:1 배정표는 데이터 생성 파이프라인 내부에만 존재** — 클라이언트 JSON·화면·공유 카드·공개 문서에는 가공명만 도달. 공개 저장소에 배정표 파일을 커밋하지 않는다 `[파이프라인 산출물에서 제외]`
2. **배정은 무작위 + 수동 검수** — 결정적 변환 함수(실명에서 가공명을 유도) 금지. 변환 규칙의 존재 자체가 역추적(연상) 경로가 되기 때문
3. **선수 ID는 등번호가 아닌 일련번호** — 등번호+포지션 조합(예: 7번 FW)은 특정 선수 연상 리스크
4. **능력 프로파일은 다출처 대조 후 직접 산정** — 특정 사이트 능력치의 복제·체계적 추출 금지 (P10 DB권). 산정 기준은 데이터 생성 노트에 기록

## Consequences

- (+) 화면·공유물·공개 문서 어디에도 연상 경로가 없음 — 실격 리스크 구조적 차단
- (+) 스키마에 실명 필드가 없어(데이터_설계 §2.1) 실수로도 실명이 클라이언트에 못 들어감
- (−) 배정표가 공개 저장소에 없으므로 재현성이 제한됨 — 수동 검수 기록을 데이터 생성 PR 본문에 남겨 감사 추적으로 보완
- (−) 무작위 배정이라 "이 가공 선수가 누구인지" 사용자가 알 수 없음 — **의도된 결과** (그것이 곧 침해 차단). 리플레이 서사는 "누가"가 아니라 "몇 분에 무슨 일이"로 성립
