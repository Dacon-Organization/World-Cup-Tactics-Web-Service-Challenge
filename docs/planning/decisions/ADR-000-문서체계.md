# ADR-000. 기획 문서 체계 — 버전 폴더 + 기능별 분리 + 융합 템플릿

- **Status**: Accepted (2026-07-22)
- **관련**: 전역 운영 규칙 §7.1, [마스터인덱스](../마스터인덱스.md)

## Context

초기 상세 설계는 `docs/planning/` 평면 구조에 기능 16종을 단일 문서(기능명세.md)로 담았다.
선행 팀 프로젝트(gc-dating-app) 회고에서 두 가지 피드백이 확인됨:
① 기능별 문서 분리 + 버전 폴더 관리가 변경 추적에 유리했다
② 예외 처리를 상세히 쓰지 않으면 구현 단계에서 "실패했는데 다음 화면으로 넘어가는" 류의 버그를 낳는다.
추가로 외부 검증 방법론(스타 수·활성도·원문 확인, 2026-07-22)을 대조했다:
github/spec-kit ⭐123k(Spec-Driven Development) · EARS(요구사항 문장 표기법, AWS Kiro 채택) ·
ADR ⭐16k(Nygard) · Gherkin Given-When-Then.

## Decision

1. **폴더 체계**: 도메인별 `version{X.Y}` 폴더 + `_changes/` 변경 누적 + `마스터인덱스.md` 대시보드 (gc-dating-app 체계, 전역 §7.1 정합). `_archive`는 v2.0 발생 시 도입.
2. **기능별 문서 분리**: F01~F16 개별 파일 + 인덱스(요약본). 에이전트 협업 시 파일 단위 diff·수정 격리 이점.
3. **융합 템플릿**: gc-dating-app 7-Step 뼈대 × spec-kit(User Story 3요소·ID 체계·`[NEEDS CLARIFICATION]`·체크리스트) × EARS(특히 IF-THEN Unwanted behaviour ≥3/기능) × GWT 수용 기준.
4. **분량이 아닌 커버리지**: 250p급 문서는 다팀·수개월 조정 비용의 산물 — 12일·1인+에이전트 프로젝트에서는 기능×EARS 요구×GWT 수용 기준 매트릭스 완비를 품질 기준으로 삼는다.
5. **research 폴더는 재편하지 않음**: 원장·findings의 상대 링크가 기획서 역추적 증거망이므로 링크 보존 우선.

## Consequences

- (+) 변경이 잦아져도 기능 단위로 diff·버전 추적 가능. 예외 누락이 문장 구조(IF-THEN)상 차단됨
- (+) `[NEEDS CLARIFICATION]` 태그로 추측 설계가 가시화되어 구현 전 게이트 가능
- (−) 문서 수 증가(16+α) → 상호 링크 관리 비용 발생. 마스터인덱스가 단일 진입점으로 이를 흡수
- (−) 버전업 시 3종 세트(전체·요약본·변경사항) 생성 의무 — 마일스톤 단위 일괄로 부담 최소화 (§7.1)
