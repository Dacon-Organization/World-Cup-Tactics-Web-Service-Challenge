# P4 — 브라우저 온디바이스 ML 추론 런타임 비교

| 항목 | 내용 |
|---|---|
| **웨이브** | W0 (기반) |
| **트리거** | T4 기술 미검증 |
| **블로킹 대상** | ONNX Runtime Web vs TF.js 결정, 번들 예산, 초기 로딩 전략 |
| **실행 순서** | 5번째 |

> **왜 필요한가**: "LLM API 없이 진짜 ML을 브라우저에서 돌린다"가 이 프로젝트의 기술적 승부수입니다.
> 그런데 이게 **실제로 잘 돌아가야** 완성도 25점을 지킵니다.
> 심사자가 링크를 열었는데 로딩에 10초가 걸리면, 아무리 좋은 모델이어도 평가는 끝납니다.
> 번들 크기와 초기 로딩 시간은 **실측 근거**로 예산을 잡아야 합니다.

## 프롬프트 (복사해서 사용)

```
[역할]
당신은 프론트엔드 성능과 온디바이스 머신러닝 배포를 다루는 기술 리서처다.

[배경]
이 Space의 프로젝트 배경을 전제로 한다. 우리는 Python으로 학습한 소형 모델
(GBDT 계열 또는 파라미터 수만~수십만 규모의 MLP)을 브라우저에서 추론하려 한다.
LLM API를 쓰지 않는 이유는 심사자가 API 키 없이 전체 기능을 확인할 수 있어야 하기 때문이다.

요구사항:
- Next.js 15 App Router 환경, Vercel 배포
- 사용자가 선수를 드래그할 때마다 추론이 재실행되므로 지연이 체감되면 안 된다
- 모바일 브라우저에서도 동작해야 한다 (심사자가 모바일로 열 가능성)
- 초기 로딩이 길면 이탈한다

[질문]
1. 브라우저 ML 추론 런타임을 다음 기준으로 비교하라:
   ONNX Runtime Web, TensorFlow.js, 그리고 Hugging Face Transformers.js.
   비교 기준: 번들 크기(gzip 기준 KB), 초기화 시간, 추론 지연,
   지원 백엔드(WASM/WebGL/WebGPU), 지원 연산자 범위, 모바일 브라우저 호환성, 유지보수 활성도.
   Transformers.js는 내부적으로 ONNX Runtime Web을 사용하는 것으로 알려져 있는데,
   그렇다면 ONNX Runtime Web을 직접 쓰는 것 대비 어떤 이득과 오버헤드가 있는가?
   우리처럼 트랜스포머가 아닌 소형 모델을 다루는 경우에도 유용한가?
2. 우리처럼 소형 모델(수백 KB 이하)을 다루는 경우,
   두 런타임 외에 더 가벼운 선택지가 있는가?
   예: 모델을 순수 JavaScript/TypeScript로 직접 구현, WASM 직접 컴파일,
   microTVM/tinygrad 계열, GBDT 전용 경량 추론 라이브러리
   특히 LightGBM/XGBoost 같은 트리 모델을 브라우저에서 추론하는
   가장 가벼운 방법이 무엇인지 구체적으로 조사하라.
3. 트리 기반 모델(LightGBM 등)과 신경망 중, 브라우저 추론 관점에서
   어느 쪽이 유리한가? 모델 크기, 추론 속도, 변환 난이도를 기준으로 답하라.
4. 초기 로딩 최적화 전략을 조사하라:
   모델 파일의 지연 로딩(lazy loading), 스트리밍, 캐싱(Cache API/IndexedDB),
   Web Worker에서의 추론 실행, WASM 파일의 CDN 전략.
   Next.js App Router 환경에서 주의할 점을 포함하라.
   또한 학습한 모델 파일을 어디에 호스팅할지 비교하라:
   (a) Next.js의 public/ 디렉터리에 넣어 Vercel에서 직접 서빙
   (b) Hugging Face Hub에 업로드해 CDN으로 로드
   (c) 별도 오브젝트 스토리지
   각각의 CORS 설정, 대역폭 제한, 캐싱 동작, 그리고
   "심사자가 별도 키 없이 접근 가능한가"를 기준으로 판단하라.
   Vercel 무료 플랜의 배포 파일 크기 제한과 대역폭 제한도 확인하라.
5. Web Worker에서 몬테카를로 시뮬레이션(수천~수만 회 반복)을 실행할 때의
   현실적인 성능 기대치는 어느 정도인가?
   메인 스레드를 막지 않으면서 결과를 점진적으로 표시하는 패턴을 조사하라.
6. 위를 종합해 우리 요구사항에 대한 권고안을 제시하라.
   런타임 선택, 모델 형식, 로딩 전략, 그리고 지켜야 할 성능 예산
   (초기 로딩 몇 초 이내, 추론 몇 ms 이내)을 구체적 수치로 제안하라.

[산출물 형식]

| 런타임/방법 | 번들 크기(gzip) | 초기화 시간 | 추론 지연 | 지원 백엔드 | 모바일 호환 | 트리모델 지원 | 신경망 지원 | 최신 릴리스 | 출처 URL |
```

## 회수 후 확인 사항

- [ ] 번들 크기가 **gzip 기준 수치**로 나왔는가
- [ ] 성능 수치에 **측정 환경**이 병기되어 있는가
- [ ] 트리 모델(LightGBM)의 브라우저 추론 방법이 구체적으로 나왔는가
- [ ] 권고안에 **구체적 성능 예산**(초/ms)이 제시되었는가

## 이 결과가 결정하는 것

| 결정 항목 | 영향 |
|---|---|
| 런타임 선택 | 전체 ML 파이프라인의 출력 형식 (ONNX vs TF.js vs 순수 JS) |
| 모델 형식 | P3에서 고른 알고리즘이 변환 가능한지 재확인 |
| 성능 예산 | 기획서 "기술 스택" 절에 명시할 목표치 |

> 💡 **주목할 가능성**: 우리 모델은 매우 작습니다. 이 경우 **ONNX Runtime Web의 런타임 자체(수백 KB)가
> 모델보다 무거워지는 역전**이 일어날 수 있습니다. 그렇다면 트리 모델을 순수 TypeScript로
> 직접 구현하는 편이 더 빠르고 가벼울 수 있습니다. 질문 2가 이걸 노린 것입니다.
> 이 결과에 따라 기술 스택이 바뀔 수 있으니, **P4 회수 전에는 ONNX를 확정으로 쓰지 않습니다.**
