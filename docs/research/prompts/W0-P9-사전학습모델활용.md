# P9 — 사전학습 모델 활용 및 파인튜닝 가능성 (Hugging Face 생태계 중심)

| 항목 | 내용 |
|---|---|
| **웨이브** | W0 (기반) |
| **트리거** | T2 대안 교착 + T3 규정 리스크 |
| **블로킹 대상** | "처음부터 학습" vs "사전학습 모델 파인튜닝" 노선 결정 |
| **실행 순서** | 5번째 (P3 직후, P4 직전) |

> **왜 별도 프롬프트인가**: P3(모델링 기법)에 욱여넣으면 질문이 10개를 넘어 각 답이 얕아집니다.
> 사전학습 모델 활용은 **데이터 조달·모델 선정·라이선스·브라우저 배포**에 모두 걸치는
> 독립된 의사결정이므로 전용 조사가 필요합니다.

## 전제

**가져온 모델을 우리 프로젝트에 맞게 커스터마이징(파인튜닝)하는 것을 전제로 조사합니다.**
"그대로 가져다 쓴다"가 아니라 "베이스로 삼아 우리 데이터로 적응시킨다"는 시나리오입니다.

이 전제가 중요한 이유:
- 파인튜닝하면 모델의 동작을 이해하고 통제하게 되므로, 기획서에서 **"왜 이렇게 예측하는가"를 설명할 수 있습니다** (기획/구현 일관성 20점)
- 반대로 **라이선스는 파인튜닝으로 세탁되지 않습니다.** 파생 모델은 베이스 모델의 라이선스를 상속합니다. 이게 이 조사에서 가장 중요한 리스크 항목입니다

## 프롬프트 (복사해서 사용)

```
[역할]
당신은 머신러닝 모델 허브 생태계와 전이학습을 다루는 기술 리서처다.

[배경]
이 Space의 프로젝트 배경을 전제로 한다. 우리는 축구 전술 시뮬레이터에 필요한 예측 모델을
구축해야 하는데, 두 가지 노선이 있다:
(A) 처음부터 소형 모델을 직접 학습
(B) 공개된 사전학습 모델을 가져와 우리 데이터로 파인튜닝/커스터마이징

노선 (B)를 진지하게 검토하려 한다. 단, 다음 제약이 있다:
- 최종 모델은 브라우저에서 추론된다 (모델 크기와 연산량이 작아야 함)
- 학습 데이터가 작다 (수백~수천 경기 수준일 가능성)
- 공개 배포되지만 수익을 발생시키지 않는 서비스다
- 대회 규칙상 라이선스 위반은 실격 사유다

[질문]
1. Hugging Face Hub에 축구/스포츠 관련 모델과 데이터셋이 어느 정도 존재하는가?
   경기 결과 예측, xG, 선수 평가, 스포츠 통계 관련 모델과 데이터셋을 검색해
   실제로 쓸 만한 것이 있는지 조사하라. 다운로드 수와 최종 업데이트 시점을 포함하라.
   품질이 낮거나 검증되지 않은 것들뿐이라면 그 사실을 명확히 밝혀라.

2. 작은 표 형식(tabular) 데이터에 적합한 사전학습 파운데이션 모델을 조사하라.
   TabPFN 계열, 그 외 tabular foundation model, 시계열 파운데이션 모델
   (Chronos, TimesFM, Moirai 등)이 우리처럼 표본이 작은 문제에
   실제로 도움이 되는가? 보고된 성능과 함께 답하라.
   특히 "수백~수천 행 규모"에서 처음부터 학습한 GBDT 대비 우위가 있는지 밝혀라.

3. 위 모델들의 라이선스를 조사하라. 이것이 이 질문 세트에서 가장 중요하다.
   각 모델의 라이선스(Apache-2.0, MIT, CC-BY-NC, RAIL 계열, 커스텀 등)와,
   그 라이선스 하에서 다음이 각각 허용되는지 구분해서 답하라:
   (a) 파인튜닝하여 파생 모델을 만들기
   (b) 파생 모델을 공개 웹서비스에 탑재하기
   (c) 파생 모델의 가중치를 공개 저장소에 배포하기
   (d) 출처 표기(attribution) 의무의 구체적 형태
   "파인튜닝하면 라이선스가 리셋된다"는 오해가 흔하므로,
   파생 모델의 라이선스 상속 원칙을 명확히 설명하라.

4. Hugging Face Hub의 데이터셋 라이선스 표기가 신뢰할 만한가?
   업로더가 원출처의 데이터를 재업로드하면서 라이선스를 잘못 표기하는
   사례가 알려져 있는가? 데이터셋 카드의 license 필드가
   실제 원출처 라이선스와 다를 수 있는 위험을 어떻게 검증해야 하는가?
   축구 데이터셋의 경우 원출처(StatsBomb, FBref 등)까지 추적하는 방법을 제시하라.

5. 파인튜닝한 모델을 브라우저 추론용으로 변환할 때의 현실적 제약은 무엇인가?
   파운데이션 모델은 대체로 크기가 크다. 우리가 검토 중인 모델들의
   실제 파라미터 수와 ONNX 변환 후 예상 파일 크기를 제시하라.
   양자화(quantization)나 증류(distillation)로 줄일 수 있는 범위와
   그때의 성능 손실을 함께 밝혀라.

6. 위를 종합할 때, 우리 상황에서 노선 (A) 직접 학습과 (B) 파인튜닝 중
   무엇을 권고하는가? 절충안(예: 파운데이션 모델로 피처를 추출하고
   가벼운 헤드만 직접 학습)이 있다면 제시하라.
   권고의 근거를 모델 성능, 라이선스 리스크, 브라우저 배포 가능성,
   그리고 "기획서에서 설명 가능한가"의 네 축으로 정리하라.

[산출물 형식]
질문 1~3에 대해 다음 컬럼의 표로 통합해서 답하라:
| 모델/데이터셋명 | 허브 | 용도 | 파라미터수/크기 | 라이선스 | 파인튜닝 허용 | 서비스 탑재 허용 | 파생 배포 허용 | 출처표기 의무 | 다운로드수 | 최종 업데이트 | URL |

"파인튜닝 허용" 등 3개 컬럼은 반드시 다음 중 하나로 표기하라:
허용 / 조건부 허용 / 금지 / 미확인
"조건부 허용"인 경우 조건을 각주로 명시하라.

질문 4는 서술형으로 답하되, 검증 절차를 단계별 체크리스트로 제시하라.
질문 5는 다음 컬럼의 표로 답하라:
| 모델 | 원본 크기 | ONNX 변환 후 | 양자화 후 | 성능 손실 | 브라우저 실행 가능 여부 |

질문 6은 결론을 먼저 쓰고 근거를 네 축으로 나누어 서술하라.

[출처 요구]
- 모델·데이터셋은 Hugging Face Hub의 실제 페이지 URL을 제시하라.
- 라이선스는 모델 카드의 표기뿐 아니라 원 라이선스 전문 URL을 함께 제시하라.
  모델 카드의 표기와 원문이 다르면 그 사실을 지적하라.
- 성능 수치는 평가 데이터셋과 함께 제시하라.

[최신성]
- 2025-01 이후 정보를 우선하라. 모델 허브는 변화가 빠르다.
- 모델의 최종 업데이트 시점을 반드시 확인하라.
  2년 이상 방치된 모델은 그 사실을 명시하라.

[배제]
- 다운로드 수가 많다는 이유만으로 품질을 추정하지 마라.
- 모델 카드의 자기 주장(성능, 용도)을 검증 없이 사실로 인용하지 마라.
  독립적 벤치마크나 논문이 없으면 [추정]으로 라벨링하라.
- 학습 데이터 출처가 밝혀지지 않은 모델은 그 사실을 명시적으로 경고하라.
  출처 불명 데이터로 학습된 모델은 라이선스 리스크를 판단할 수 없다.
- "일반적으로 파인튜닝은 괜찮다" 같은 서술을 배제하라. 라이선스 조항 근거를 요구한다.

[언어]
한국어. 모델명·라이선스명은 원문 병기.
```

## 회수 후 확인 사항

- [ ] 파생 모델의 **라이선스 상속 원칙**이 조항 근거와 함께 설명되었는가
- [ ] HF 데이터셋 라이선스 표기의 **신뢰성 검증 절차**가 나왔는가
- [ ] 후보 모델의 **ONNX 변환 후 크기**가 구체적 수치로 나왔는가
- [ ] 노선 (A)/(B) 권고가 **네 축 모두**에 대해 근거를 갖추었는가

## 이 결과가 결정하는 것

| 결과 | 후속 영향 |
|---|---|
| 쓸 만한 사전학습 모델 **있음 + 라이선스 문제없음 + 브라우저 실행 가능** | 노선 (B) 채택. P3의 알고리즘 선정을 파인튜닝 관점으로 재조정 |
| 모델은 있으나 **크기가 브라우저 예산 초과** | 절충안 검토 — 파운데이션 모델로 피처 추출 후 경량 헤드만 배포 |
| 모델은 있으나 **라이선스가 NC(비상업)** | ⚠️ 무료 서비스라도 조건 확인 필요. 원장에 긴급 적재 |
| 쓸 만한 것이 **없음** | 노선 (A) 직접 학습 확정. 다만 **HF 데이터셋은 별도로 활용 가능** |

## 교차 검증 지점

| 대조 | 확인할 것 |
|---|---|
| **P9 ↔ P4** | P9가 추천한 모델이 P4의 성능 예산(번들 크기·추론 지연) 안에 들어오는가 |
| **P9 ↔ P2** | HF 데이터셋의 원출처가 P2에서 조사한 라이선스와 일치하는가 |
| **P9 ↔ P3** | 파인튜닝 노선이 P3의 기법 선정을 무효화하는가, 보완하는가 |

> ⚠️ **가장 경계할 시나리오**: 파운데이션 모델이 성능은 좋은데 크기가 100MB를 넘어
> 브라우저에서 못 돌리는 경우입니다. 이때 "성능이 좋으니까 서버를 쓰자"로 가면
> **API 키 없이 확인 가능해야 한다는 대회 규칙**과 충돌합니다.
> P9와 P4를 반드시 함께 놓고 판단해야 하는 이유입니다.
