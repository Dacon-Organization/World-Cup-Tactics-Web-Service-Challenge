### 순서에 이유가 있습니다

- **P2가 1순위**인 이유: 데이터를 쓸 수 있는지가 나머지 전부의 전제입니다. 여기서 "못 쓴다"가 나오면 P3·P4의 모델 설계 전제가 바뀝니다.
- **P6이 2순위**인 이유: 리플레이 모드의 출발점 데이터입니다. 여기가 틀리면 서비스 신뢰가 무너집니다.
- **P3 → P9 → P4 순서**인 이유: 어떤 알고리즘이 필요한지 알아야(P3) 그걸 대신할 사전학습 모델이 있는지 물을 수 있고(P9), 무엇을 쓸지 정해져야 그게 브라우저에서 돌아가는지 물을 수 있습니다(P4).

### 교차 검증이 필요한 지점

| 대조 항목 | 출처 | 확인할 것 |
|---|---|---|
| 학습 데이터 규모 | P2 ↔ P3 | P3가 요구하는 데이터량을 P2로 확보 가능한가 |
| 알고리즘 ↔ 런타임 | P3 ↔ P4 | P3에서 고른 모델이 P4의 런타임으로 변환 가능한가 |
| **모델 크기 ↔ 성능 예산** | **P9 ↔ P4** | **P9가 추천한 사전학습 모델이 브라우저 예산 안에 들어오는가** |
| **허브 데이터 ↔ 원출처** | **P9 ↔ P2** | **HF 데이터셋의 라이선스 표기가 원출처와 일치하는가** |
| 추론 지연 ↔ 인지 임계값 | P4 ↔ P5 | 추론이 사용자가 느끼기에 충분히 빠른가 |
| 차별화 주장 ↔ 실제 구현 가능성 | P1 ↔ P3·P4 | 우리가 내세울 차별점이 기술적으로 실현 가능한가 |

> ⚠️ **P9 ↔ P4가 가장 위험한 교차점입니다.** 파운데이션 모델이 성능은 좋은데
> 크기가 브라우저 예산을 넘으면, "그럼 서버를 쓰자"는 유혹이 생깁니다.
> 그건 **"심사자가 별도 키 없이 확인 가능해야 한다"는 대회 규칙과 충돌**합니다.
> 두 결과를 반드시 함께 놓고 판단하세요.

교차 검증에서 충돌이 발견되면 **원장에 기록**하고 어느 쪽을 채택했는지 근거를 남깁니다.
