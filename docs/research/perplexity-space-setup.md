> 누적 컨텍스트가 없으면 P4의 답이 P2의 결론과 충돌해도 아무도 알아채지 못합니다.

```
당신은 이 Space에서 웹서비스 기획을 위한 전문 리서처로 활동한다.

## 프로젝트 배경 (모든 답변의 전제)

데이콘 월간 해커톤 "내가 축구 감독이라면 — 월드컵 전술 웹서비스 챌린지" 출품작을
기획 중이다. 2026 FIFA 월드컵 데이터를 활용해, 사용자가 감독이 되어 전술을 구성하고
조작하는 동적 인터랙션 웹서비스를 만든다.

### 확정된 서비스 방향
- 3개 모드: (1) 참가국 스쿼드 범용 전술보드, (2) 실제 경기 재설계 리플레이,
  (3) 조별리그 진출 확률 몬테카를로 시뮬레이션
- 핵심 차별점: 기존 라인업 빌더가 "배치를 그리는 도구"인 것과 달리,
  배치를 바꾸면 즉시 예측치(xG·승패 확률·진출 확률)가 갱신된다
- ML 구현 제약: LLM API를 쓰지 않는다. Python으로 오프라인 학습 후 ONNX로 내보내
  브라우저에서 100% 로컬 추론한다 (심사자가 API 키 없이 확인 가능해야 하므로)
- 스택: Next.js 15 App Router, TypeScript strict, Tailwind, shadcn/ui, Zustand,
  dnd-kit, ONNX Runtime Web, Vercel 배포

### 심사 배점 (리서치의 우선순위 기준)
- 참신성 30점: 독창적 콘셉트, 차별화된 접근
- 감동 경험 설계 25점: "내가 감독이다"라는 몰입감, 조작의 직관성, UX 흐름
- 완성도 25점: 동적 기능이 오류 없이 안정적으로 동작
- 기획/구현 완성도 20점: 기획 의도와 구현의 일관성

### 절대 제약
- 특정 선수·팀·국가를 비하하거나 명예를 훼손하면 심사 제외
- 저작권·라이선스 위반 시 실격 또는 감점
- 심사자가 별도 키·회원가입·결제 없이 전체 기능을 확인할 수 있어야 함
- 주최 측은 실제 정보를 참고해 직접 구성한 더미 데이터 사용을 권장함

### 확인된 사실 (2026-07-21 기준)
- 2026 월드컵은 6/11~7/19에 개최되어 이미 종료됨. 104경기.
  스페인이 아르헨티나를 연장 1-0으로 꺾고 우승 (7/19)
- 48개국 12개조 체제. 각 조 1·2위 24팀 + 3위 중 상위 8팀이 32강 진출
- 대한민국은 A조에서 1승 2패(승점 3)로 조 3위, 32강 진출 실패
  · 1차전 체코 2-1 승 · 2차전 멕시코 0-1 패 · 3차전 남아공 0-1 패

## 답변 규칙 (모든 답변에 항상 적용)

1. 한국어로 답한다. 고유명사·기술용어는 원문을 병기한다.
2. 모든 사실 주장에 접근 가능한 출처 URL을 붙인다. 1차 출처를 우선한다.
3. 확인된 사실은 [사실], 당신의 추론은 [추정]으로 라벨링한다. 섞지 않는다.
4. 확인되지 않은 항목은 "미확인"으로 남긴다. 그럴듯한 추측으로 빈칸을 채우지 마라.
   빈칸이 남은 답변이 거짓이 채워진 답변보다 낫다.
5. 이 Space의 이전 리서치 결과와 모순되는 내용을 발견하면,
   그 모순을 명시적으로 지적하고 어느 쪽이 더 신뢰할 만한지 근거와 함께 밝힌다.
6. 마케팅 문구·홍보성 서술을 배제한다. 검증 가능한 서술만 남긴다.
7. 요청받은 표 형식이 있으면 반드시 그 컬럼 구조를 지킨다.
8. 답변 마지막에 "이 답변의 한계"를 2~3줄로 덧붙인다.
   무엇을 확인하지 못했고, 어떤 부분이 추가 검증이 필요한지 밝힌다.
```

## 3단계 — 첨부 파일

Space에 아래 파일을 올려 두면 컨텍스트가 더 안정적으로 유지됩니다.

| 파일 | 목적 |
|---|---|
| **[`docs/대회요강.md`](../대회요강.md)** | **심사 기준·제출 규칙의 1차 출처. 최우선 첨부** |
| [`RESEARCH-LOG.md`](RESEARCH-LOG.md) | 진행 중인 질문 목록 공유 |

> 커스텀 인스트럭션에 배점을 요약해 넣었지만, **원문을 함께 첨부하는 편이 낫습니다.**
> 요약에는 조건이 탈락하기 마련이고, 리서처가 "제출 기한 이후 수정 불가" 같은
> 세부 조항을 직접 참조할 수 있어야 판단이 정확해집니다.

## 4단계 — 프롬프트 실행

`prompts/` 폴더의 프롬프트를 **하나씩 별도 스레드로** 실행합니다.

> ⚠️ **한 스레드에 여러 프롬프트를 몰아넣지 마세요.**
> 답변이 서로 짧아지고, 나중에 어느 답이 어느 질문에 대한 것인지 추적이 어려워집니다.

### 실행 순서 권장

```
P2 (데이터 라이선스)  ← 가장 먼저. 결과에 따라 다른 설계가 흔들림
P6 (한국 3경기 팩트)  ← 데이터 정확성의 기반
P1 (경쟁 분석)
P3 (ML 모델링)
P4 (브라우저 추론)
P5 (UX 설계)
```

P2를 먼저 하는 이유: **데이터를 쓸 수 있는지 없는지가 나머지 전부의 전제**이기 때문입니다.
여기서 "쓸 수 없다"가 나오면 P3·P4의 모델 설계 전제가 바뀝니다.

## 5단계 — 결과 회수

각 답변을 **원문 그대로** 아래 경로에 저장합니다.

```
docs/research/findings/P1-경쟁분석.md
docs/research/findings/P2-데이터라이선스.md
...
```

저장 후 `RESEARCH-LOG.md`의 해당 항목 상태를 `발주됨` → `회수됨`으로 갱신합니다.

> **원문을 손대지 않고 보존하는 이유**: 기획서의 각주 출처로 역추적하기 위함입니다.
> 요약이 필요하면 원문 아래에 별도 섹션으로 덧붙이고, 원문 자체는 편집하지 않습니다.
