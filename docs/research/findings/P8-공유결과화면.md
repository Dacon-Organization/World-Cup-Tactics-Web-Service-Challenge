## 요약 및 판단

### Q1. 검증된 공유 패턴 — 공유를 유발하는 5요소 (채택)

**Wordle 이모지 그리드** (1차 출처: 창작자 Josh Wardle 트윗 2021-12-16 "spoiler-free emoji grid"):

| # | 요소 | 근거 |
|---|---|---|
| ① | **스포일러 프리** — 결과의 구조만 노출, 내용(정답)은 비노출 | Wardle 트윗 원문. 받는 사람의 경험을 훼손하지 않아 공유 죄책감 제거 |
| ② | **드라마의 압축** — 시도·실패·성공의 서사(arc)가 한 장의 그래픽 시놉시스로 | 게임 철학자 Thi Nguyen(유타대): "huge amount of information - and drama - packed into that little graph" (Smithsonian Mag 재인용) |
| ③ | **공통 퍼즐 → 비교 가능성** — 모두가 같은 문제를 풀기에 결과가 비교·경쟁·공감 대상이 됨 | 심리학자 Lee Chambers: 커뮤니티감("얼마나 어려웠나") + 경쟁("나는 몇 번 만에") (Insider/Smithsonian) |
| ④ | **플랫폼 독립 텍스트** — 이모지 그리드는 이미지가 아니라 텍스트 → 어디든 붙여넣기 가능, 생성 비용 0 | Wardle 트윗 형식 자체 |
| ⑤ | **관찰의 제품화** — 사용자(NZ의 Elizabeth S.)가 수작업으로 하던 공유를 버튼 한 개로 제품화 | Forbes 보도. "공유 기능은 발명이 아니라 관찰" |

**Spotify Wrapped**: 공식 뉴스룸(2025-12) — "Every data story includes a share card". **데이터 스토리 1개당 공유 카드 1개**가 설계 원칙. 개인화된 결과 = 정체성 표현 수단.

**우리 서비스 적용 (채택)**:
- 시뮬레이션 결과 화면에 **"내 전술 카드"** 1:1 대응 (Wrapped 원칙): 포메이션 미니맵 + 슬라이더 설정 + 승/무/패 확률 밴드.
- **이모지 그리드형 텍스트 공유** (Wordle ①④): P12 채택 포맷인 10×10 아이콘 배열을 텍스트로 번역 — 예: 승률 62%를 `🟩×6 🟨×2 ⬜×2` 축약 행렬로. 스코어 스포일러 없이 "내 전술의 확률 구조"만 노출.
- **공통 퍼즐 구조** (Wordle ③): 리플레이 모드의 한국 3경기는 모두에게 동일한 "문제" → "같은 멕시코전, 다른 전술" 비교가 Wordle의 공통 퍼즐과 동형. **비교 가능성이 공유 동기의 핵심**이므로 공유 카드에 경기 식별자(상대 국기+라운드)를 반드시 포함.

### Q2. 기술 구현 — 공유 3계층 (채택)

| 계층 | 구현 | 근거(원문 확인) |
|---|---|---|
| **1. URL + 동적 OG 이미지** (기본) | 전술 상태를 URL 파라미터로 인코딩 + Next.js `opengraph-image.tsx`(`ImageResponse`, satori 내장)로 1200×630 카드 서버 생성 | Next.js 공식 문서(v16, 2026-03): 파일 컨벤션·동적 `params`·정적 최적화 기본. Vercel 배포와 정합, **심사자 키 불필요** |
| **2. Web Share API** (모바일 강화) | `navigator.share()` — OS 네이티브 공유 시트 | MDN+web.dev: Android Chrome 93+/iOS Safari 12.1+/macOS Safari 12.1+/Chrome·Edge 93+. **Firefox 데스크톱은 플래그 뒤** → `navigator.share` **feature detection 필수**. HTTPS + 사용자 제스처(transient activation) 요건 |
| **3. 클립보드 텍스트** (전 브라우저 폴백) | 이모지 그리드 `navigator.clipboard.writeText` | Wordle 방식 그대로. 계층 2 미지원 시 자동 폴백 |

**클라이언트 캡처 라이브러리 대조 (gh api, 2026-07-22)**:

| 레포 | 스타 | 라이선스 | 판정 |
|---|---|---|---|
| `vercel/satori` | 13.7k★ | MPL-2.0 | `next/og`에 내장 → **직접 의존 불필요** |
| `bubkoo/html-to-image` | 7.2k★ | MIT | 클라이언트 캡처 필요 시 **1순위** (2026년에도 활발) |
| `niklasvh/html2canvas` | 31.9k★ | MIT | 2024-07 이후 정체 → 차선 |
| `tsayen/dom-to-image` | 10.8k★ | **NOASSERTION** | 라이선스 불명확+2024 이후 정체 → **기각** |
| `nygardk/react-share` | 2.8k★ | MIT | SNS별 버튼 필요 시에만 |

→ **채택: 서버 생성(OG) 우선, 클라이언트 캡처는 html-to-image 한정.** "이미지 저장" 버튼이 필요해지면 html-to-image로 결과 카드 DOM을 PNG화.

### Q3. 카카오톡 — 앱 키 없이 성립 (채택)

- 카카오톡 링크 미리보기는 **OG 태그 자동 스크래핑**으로 성립 (카카오 공식 devtalk: 스크랩은 **80/443 포트만 허용**, `og:title`/`og:image` 기반). → **Kakao SDK·앱 키 없이** 계층 1(URL+OG)만으로 카톡 공유 카드 완성. 대회 규칙 "심사자가 별도 키 없이 확인 가능"과 정합.
- Kakao SDK 템플릿 메시지 공유는 앱 키 필요 → **기각**.
- ⚠️ 카톡은 스크랩 결과를 **캐시**함(공식 확인) → OG 이미지 변경 시 [카카오 공유 디버거](https://developers.kakao.com/tool/debugger/sharing)로 캐시 갱신. 시연 전 체크리스트에 포함.

### Q4·Q5. 규정 리스크 (P7 전면 적용)

- 공유 카드·OG 이미지에도 **P7 에셋 가이드 그대로 적용**: 실명·이니셜 금지(가공명), FIFA IP 금지, 국기+색상 팔레트+자체 SVG만.
- OG 이미지에 폰트 임베드 시: 서버 렌더링 산출물은 이미지(PNG)이므로 폰트 파일 재배포 아님 → OFL상 문제 없음. 저장소에 폰트 파일을 두는 경우만 LICENSE 동봉(P7 결론 유지).
- **비하 금지(카피 단위)**: 공유 카드 문구도 "다른 선택지" 관점 — "감독님의 선택은 달랐다" O / 특정 인물 판단을 "잘못"으로 규정 X.
