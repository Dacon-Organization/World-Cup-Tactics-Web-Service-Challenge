## 요약 및 판단

### Q1. 카카오 스크랩 캐시 TTL = **1시간 (공식 답변 확보)** 🆕

- 카카오 공식 답변자(tim.l) 원문: **"기본적으로 같은 url에 대해 1시간 캐시 됩니다."**
  (데브톡 topic 120788 "미리보기 캐시 저장 기간 관련 문의")
- 강제 갱신 도구: [카카오 OG 캐시 초기화](https://developers.kakao.com/tool/clear/og)
  (Scrap FAQ topic 22238 "미리보기(스크랩) 캐시 삭제 기능").
- **P8 체크리스트 구체화**: 시연 전 OG 이미지를 바꿨다면 ① 캐시 초기화 도구 실행
  또는 ② 1시간 경과 대기. **URL 쿼리가 다르면 다른 캐시 키** → 전술 상태별 URL은
  각자 신규 스크랩(캐시 충돌 없음).

### Q2. URL 길이 한계 — **실무 안전선 2,000자, 우리 설계 여유 20배 이상**

| 계층 | 한계 (출처) |
|---|---|
| Chrome/Chromium | **2MB** (크로미엄 문서 — IPC DoS 방지 목적), 주소창 표시는 32kB |
| Safari | 약 80,000자 |
| Firefox | 사실상 무제한, 주소창 표시 ~65,536자 |
| 레거시·관행 | 구 IE 2,083자 · sitemap 프로토콜 2,048자 → **크로스 플랫폼 안전선 ≈ 2,000자** |
| 카카오 스크랩 대상 URL 상한 | **미문서화** `[미확인 유지]` — 안전선 내 설계로 회피 |

**전술 상태 인코딩 여유 `[설계 계산]`**: 포메이션 ID(1B) + 슬라이더 5개(각 1B) +
상대 팀 코드(3B) + 프리셋/모드 플래그(2B) ≈ 수십 바이트 → base64url ≤ 100자.
도메인+경로 포함 총 URL ≈ 150자 < 2,000자 안전선의 **1/13**. 선수 배치 좌표(22×2B)를
추가해도 ≤ 300자. **길이 한계는 설계 리스크가 아님을 확정.**

### Q3. Web Share API `files` — **iOS 15 / Safari 15부터 (공식 릴리스 노트)**

| 기능 | 지원 (출처) |
|---|---|
| `navigator.share()` 기본 | iOS Safari 12.2+ · macOS Safari 12.1+ · Chrome/Edge 89+(데스크톱, 128부터 완전) · Chrome Android ✅ |
| **`share({files})` (Level 2)** | **Safari 15 / iOS 15·iPadOS 15부터** (WebKit 공식: "Web Share level 2 — sharing files from a web page to an app". iOS 14는 플래그 뒤) |
| Firefox 데스크톱 | **전 버전 미지원** (caniuse 2~155 확인) — P8의 "플래그 뒤" 서술을 "미지원"으로 강화 |
| 공통 요건 | HTTPS + transient activation(버튼 클릭 트리거) + `canShare({files})` 사전 검사 |

**채택 재확정**: P8 공유 3계층 그대로 — 이미지 파일 공유는 `canShare` feature detection
후 시도, 실패 시 URL 공유(계층 1)·클립보드(계층 3) 폴백. iOS 15 미만은 자동으로 폴백 경로.

## 기존 결론과의 모순

- **P8과 모순 없음 — 수치 확정으로 보강.** ① "캐시 주의" → TTL 1시간 확정
  ② Firefox 데스크톱 "플래그 뒤" → caniuse 기준 미지원으로 강화(폴백 설계 영향 없음)
  ③ URL 인코딩 전략의 길이 리스크 소거.
