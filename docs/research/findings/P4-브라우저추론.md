## 요약 및 판단

### 채택한 결론
- **런타임**: **ONNX Runtime Web의 wasm 전용 서브셋(`ort.wasm.min.js`, raw-min 약 47KB) 직접 사용**. TreeEnsemble 연산자(`ai.onnx.ml`) 표준 지원. [사실]
- **Transformers.js 배제**: 내부적으로 ONNX Runtime Web을 그대로 쓰면서 트랜스포머 전용 전처리 레이어를 얹는 구조 → 트리모델·소형 MLP엔 **순수 오버헤드**. [사실]
- **TF.js 부적합**: 트리모델 공식 연산자 없음, 번들이 큼(tfjs-core gzip ~217KB + webgl +147KB). [사실]
- **모델 형식**: LightGBM/GBDT는 **onnxmltools로 ONNX(TreeEnsemble) 변환**. 더 가볍게는 **m2cgen으로 순수 JS 코드 생성**(런타임 의존성 0)이 차선책. [사실]
- **호스팅**: **Next.js `public/` + Vercel 직접 서빙** — CORS 무문제, 키 없이 접근 가능, Hobby 100GB/월로 심사 트래픽 충분. HF Hub는 대안이나 심사 당일 외부 CDN 장애 리스크. [사실+추정]
- **로딩 전략**: Service Worker Cache API로 모델·wasm 캐싱, 추론은 **Web Worker**에서(메인 스레드 비블로킹). threaded wasm(10~20MB)은 Vercel 소스 업로드 한도 영향 → 소형 모델은 non-threaded 선택.
