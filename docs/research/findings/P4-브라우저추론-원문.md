## 1. 브라우저 ML 추론 런타임 비교

[사실] ONNX Runtime Web(onnxruntime-web)은 CPU 백엔드에 WASM(WebAssembly), GPU 가속에 WebGL·WebGPU를 지원하며, 기본 전체 기능 번들인 `ort.all.min.js`는 minified 상태로 약 773KB이고 문서 자체가 "500KB 이상"이라고 명시한다. 트리모델 전용 경량 서브셋인 `ort.wasm.min.js`(WASM 전용, WebGL/WebGPU 제외)는 약 47KB(minified, gzip 아님, 이하 raw-min 기준)로 확인된다. gzip 수치는 npm/jsDelivr 배포 페이지에 별도로 게시되어 있지 않아 [미확인]이며, 위 KB는 raw-minified 기준임을 명시한다.[^1][^2]

[사실] TensorFlow.js는 `@tensorflow/tfjs-core`만 사용해도 compressed(gzip) 기준 약 217KB이고, `@tensorflow/tfjs-layers`를 포함한 전체 패키지(`@tensorflow/tfjs`)는 compressed 기준 약 208~267KB 범위로 보고되며, WebGL 백엔드를 추가하면 약 147KB가 더 붙는다(측정 환경: pnpm 의존성 트리 분석, tfjs 4.17.0). 커뮤니티 보고에 따르면 `npm run build` 시 tfjs가 앱에 50MB를 추가한 사례도 있어(번들러·트리쉐이킹 설정에 크게 의존), 실제 배포 크기는 사용하는 서브패키지와 트리쉐이킹 수준에 따라 크게 달라진다. tfjs-core만 별도로 tree-shaking하면 122KB(14%) 절감된다는 공식 사례도 있다(tfjs 1.2.7 기준, 오래된 버전).[^3][^4][^5][^6]

[사실] Transformers.js(`@huggingface/transformers`, 구 `@xenova/transformers`)는 내부적으로 ONNX Runtime을 그대로 사용한다. Node에서는 `onnxruntime-node`, 브라우저에서는 `onnxruntime-web`을 백엔드로 로드하며, 두 패키지를 모두 정적으로 import하고(동적 import가 webpack에서 동작하지 않아) 환경별로 골라 쓰는 구조다. 즉 브라우저 번들 크기 관점에서 Transformers.js는 "ONNX Runtime Web + 전처리/후처리/파이프라인 레이어"의 합이며 ONNX Runtime Web 단독보다 항상 크다.[^7][^8]

질문 1의 "Transformers.js가 ONNX Runtime Web 대비 갖는 이득/오버헤드"에 대해: [사실] 이득은 (1) 모델 로딩·전처리·후처리·토크나이징을 감싸는 고수준 `pipeline` API로 개발 편의성이 높고, (2) 양자화 모드(`fp32/fp16/q8/q4`) 전환이 옵션 하나로 가능하며, (3) Hugging Face Hub의 사전 변환된 ONNX 모델 생태계를 그대로 활용할 수 있다는 점이다. 오버헤드는 (1) Node/브라우저 양쪽 백엔드 패키지를 동시에 정적 임포트하는 구조상 불필요한 코드가 브라우저 번들에 포함될 수 있고, (2) 트랜스포머 아키텍처에 맞춰진 전처리 파이프라인(토크나이저, 이미지 프로세서 등)은 GBDT/소형 MLP 같은 비-트랜스포머 모델에는 전혀 쓸모가 없어 순수 오버헤드로만 작동한다는 점이다. [추정] 따라서 본 프로젝트(트리 모델 또는 수만~수십만 파라미터 MLP)에는 Transformers.js가 적합하지 않고, ONNX Runtime Web을 직접 쓰거나 더 가벼운 대안(2번 항목)이 유리하다.[^14][^7]

## 2. 소형 모델(수백 KB 이하)을 위한 경량 대안

[사실] m2cgen(BayesWitnesses/m2cgen)은 scikit-learn 계열 모델(선형모델, 트리, 랜덤포레스트 등)을 Java, C, JavaScript 등 네이티브 코드로 변환하며 런타임 의존성이 전혀 없는 순수 코드를 생성한다. 트리 모델의 경우 학습된 분기 조건을 if-else 트리로 그대로 JS 함수로 뽑아내므로, 런타임 라이브러리(ONNX Runtime, TF.js 등)를 전혀 로드하지 않고 수 KB짜리 순수 JS 파일 하나로 추론이 가능하다. 다만 m2cgen이 LightGBM 자체를 직접 지원하는지(주로 scikit-learn API 지원) [미확인]이므로, LightGBM Booster를 sklearn 래퍼(LGBMClassifier/Regressor)로 학습한 뒤 m2cgen에 넣는 경로가 필요하다.[^16][^17]

[사실] LightGBM/XGBoost를 ONNX로 변환하는 표준 경로는 onnxmltools의 `convert_lightgbm` 함수로, `ai.onnx.ml` 도메인의 `TreeEnsembleClassifier`/`TreeEnsembleRegressor` 연산자를 사용한 ONNX 그래프를 생성한다. 이 경로는 onnxruntime-web의 wasm 전용 서브셋(약 47KB)만으로 추론 가능하며, WebGL/WebGPU/전체 신경망 연산자 지원은 필요 없다. Hummingbird 라이브러리를 통해 트리모델을 신경망 형태(행렬연산)로 재표현하는 방식도 존재하나, 이는 GPU 가속을 노리는 방식이라 소형 모델·CPU 추론에는 불필요한 복잡성을 추가한다.[^2][^18][^19][^20][^11][^21]

가장 가벼운 방법 순위(구체적 수치가 확인된 것 우선):

- **순수 JS 코드 생성(m2cgen)**: [사실] 런타임 의존성 0, 트리 구조를 그대로 if-else로 변환, 파일 크기는 트리 개수·깊이에 비례(수십~수백 트리면 수십~수백 KB 코드 가능, 정확한 KB는 모델마다 다르므로 일반화된 수치는 [미확인]).[^17]
- **ONNX + onnxruntime-web wasm 서브셋**: [사실] 런타임 약 47KB(raw-min) + 모델 파일(트리 개수에 비례, 통상 수십~수백KB) + `ai.onnx.ml` TreeEnsemble 연산자 지원. m2cgen보다 런타임 오버헤드(47KB)가 추가되지만 모델 재학습 없이 재배포·버전관리가 쉽다.[^10][^11][^2]
- **WASM 직접 컴파일, microTVM/tinygrad 계열**: [미확인] 이번 리서치에서 브라우저向 실측 벤치마크나 통합 사례를 확인하지 못했다. microTVM은 마이크로컨트롤러/엣지 디바이스 타깃이 우선이고 브라우저 배포 경로는 별도 검증이 필요하다 [추정].
- **GBDT 전용 경량 추론 라이브러리(Treelite)**: [사실] Treelite는 트리 모델을 컴파일된 C 코드/체크포인트로 변환하는 프로젝트이나, 확인된 자료는 NVIDIA Triton FIL 백엔드(서버 사이드) 통합이 중심이며 브라우저(WASM/JS) 타깃 공식 지원 여부는 [미확인]이다.[^22]

[추정] 결론적으로, LightGBM 같은 트리 모델을 브라우저에서 가장 가볍게 추론하는 방법은 m2cgen으로 순수 JS 함수를 생성하는 것이며, 재현성·검증 편의성이 더 중요하다면 onnxmltools로 ONNX 변환 후 onnxruntime-web wasm 서브셋(약 47KB)을 쓰는 것이 차선책이다.

## 3. 트리 기반 모델 vs 신경망: 브라우저 추론 관점

[사실] 모델 크기 측면에서 트리 모델(LightGBM/XGBoost)은 트리 개수 × 트리당 노드 수에 비례해 크기가 결정되며, 수백 개 트리로도 수백 KB 이하로 유지하기 쉽다. 반면 신경망은 파라미터 수 × 4바이트(fp32) 또는 1바이트(int8 양자화)로 크기가 결정되므로, 수만~수십만 파라미터급 MLP는 fp32 기준 수백 KB, 양자화 시 수십~백여 KB로 트리 모델과 비슷한 범주에 들어올 수 있다 [추정, 정확한 벤치마크는 모델 스펙에 의존해 일반화 수치 없음].

추론 속도 측면에서 트리 모델은 각 트리를 순회하는 분기 연산(비교·분기)이 대부분이라 WASM 스칼라 연산으로도 충분히 빠르며 SIMD·GPU 가속의 이득이 크지 않다. 신경망은 행렬곱(GEMM) 연산이 핵심이라 WASM SIMD나 WebGL/WebGPU 가속의 효과가 상대적으로 크지만, [사실] ONNX Runtime Web 문서는 "모델을 ORT 포맷으로 변환하면 초기화 시간과 최대 메모리 사용량이 최적화된다"고 명시하는 정도로, 본 프로젝트 규모(소형 모델)에서는 초기화 오버헤드(런타임 로딩·세션 생성)가 실제 순수 추론 시간보다 클 가능성이 높다는 것이 배제 규칙에서 지시한 핵심 전제다.[^12]

변환 난이도 측면에서 LightGBM→ONNX 변환은 onnxmltools의 단일 함수 호출로 끝나는 성숙한 파이프라인이며, opset·zipmap·split 등 세부 옵션까지 문서화되어 있다. 신경망(PyTorch/TF→ONNX)도 `torch.onnx.export`나 TF 변환기로 비교적 표준화되어 있으나, MLP처럼 단순한 구조라면 두 경로 모두 난이도 차는 크지 않다 [추정]. 종합하면, 트리 모델은 (1) 런타임 오버헤드에 덜 민감하고, (2) TreeEnsemble 연산자만 지원하면 되므로 ONNX Runtime Web wasm 서브셋 47KB 수준으로 충분해 소형 모델 배포 관점에서 신경망보다 유리하다 [추정].[^18][^19][^11][^23]

## 4. 초기 로딩 최적화 및 호스팅 전략

[사실] Cache API와 Service Worker를 통한 정적 자산(모델 파일, WASM 바이너리) 캐싱은 브라우저 표준 기능으로, 두 번째 방문부터는 네트워크 요청 없이 즉시 로드가 가능하다. Transformers.js 공식 안내도 "모델 파일은 최초 다운로드 후 브라우저에 캐시되어, 오프라인 사용을 포함해 반복 사용 시 재다운로드가 필요 없다"고 명시하며, 프로덕션에서는 Hub 의존 대신 자체 호스팅을 권장한다. Web Worker에서 추론을 실행해 메인 스레드를 막지 않는 패턴도 동일 문서에서 "프로덕션 기능이라면 반드시 Web Worker로 옮겨야 한다"고 명시적으로 권고한다.[^14]

[사실] Next.js App Router 환경에서 주의할 점: `public/` 디렉터리에 넣은 파일은 정적 자산으로 그대로 서빙되며 별도 API 라우트 없이 CDN 캐시 대상이 되지만, ONNX Runtime Web의 wasm 파일(`ort-wasm-simd-threaded.wasm` 등)은 threaded 버전 기준 10~20MB(비압축, jsDelivr 파일 목록 기준)에 달할 수 있어 `public/`에 두면 Vercel 배포 소스 파일 업로드 한도(Hobby 100MB, Pro 1GB)에 영향을 줄 수 있다. threaded/SIMD가 필요 없는 소형 모델이라면 non-threaded `ort-wasm-simd.wasm` 계열(수백 KB~수MB대)을 선택해 크기를 줄이는 것이 실용적이다 [추정, 정확한 non-threaded 파일 크기는 이번 리서치에서 KB 단위로 확인하지 못해 미확인].[^24][^2]

## 5. Web Worker 몬테카를로 시뮬레이션 성능

[사실] 이번 리서치에서 "기기·브라우저·반복 횟수가 명시된" 몬테카를로 시뮬레이션 벤치마크 수치를 확인하지 못했으므로, 구체적 ms/iteration 수치는 [미확인]으로 남긴다. 배제 규칙에 따라 환경이 명시되지 않은 벤치마크는 인용하지 않는다.

[추정] 일반적인 패턴으로는, 순수 JS로 구현된 몬테카를로 루프(단순 확률 샘플링 + 소형 트리/MLP 추론)를 Web Worker에서 실행하면 메인 스레드 블로킹 없이 진행 상황을 `postMessage`로 스트리밍할 수 있다. 권장 패턴은 (1) 전체 반복을 배치(예: 500~1000회)로 나눠 각 배치 완료 후 중간 결과를 메인 스레드에 전송, (2) 메인 스레드는 `requestAnimationFrame` 또는 상태 업데이트로 progress bar를 갱신, (3) `Transferable`/`SharedArrayBuffer`를 사용해 배열 복사 오버헤드를 줄이는 것이다. 이 구조 자체는 Transformers.js 문서가 권고하는 "무거운 연산은 Web Worker로 옮기라"는 일반 원칙과 부합한다.[^14]

## 이 답변의 한계

ONNX Runtime Web과 TF.js의 gzip 기준 정확한 번들 크기, 초기화 시간, 추론 지연에 대해 "기기·브라우저·모델 크기가 명시된" 벤치마크를 찾지 못해 다수 항목을 [미확인]으로 남겼다. m2cgen의 LightGBM 직접 지원 여부, WASM 직접 컴파일/microTVM/tinygrad의 브라우저 실측 사례, 오브젝트 스토리지(S3/R2)의 CORS·비용 세부사항도 1차 출처로 확인하지 못했다. 성능 예산(초기 로딩·추론 지연 수치)은 실측 데이터가 아닌 설계 목표로서의 [추정]이므로, 실제 모델을 만든 뒤 반드시 Chrome DevTools Performance 탭 등으로 재측정이 필요하다.
